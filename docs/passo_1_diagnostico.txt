# Diagnóstico Atual do Fluxo de Ordens de Serviço (O.S.)

## 1. Localização no Projeto
- **Menu/Submenu:**
  - `Criar O.S. Manual` (target: `ordemServicoManual`)
  - `O.S. Escritório` (target: `ordemServicoEscritorio`)
- **Telas (em `docs/index.html` e `docs/app.js`):**
  - **Criar O.S. Manual:** `<section id="ordemServicoManual">`, lógica na função/objeto `App.actions.osManual`
  - **O.S. Escritório:** `<section id="ordemServicoEscritorio">`, lógica no objeto `App.osEscritorio`
- **Fluxo/Status Atuais:**
  - Status usados: `PLANEJADA` (padrão na criação), `EM_EXECUCAO`, `CONCLUIDA`, `DIVERGENTE`.
  - A O.S. é criada com `PLANEJADA`.
- **Lógica de Planejamento Existente:**
  - Não há um módulo de planejamento avançado com ciclos/regras no código base (aparentemente). O planejamento se resume a criar a O.S. com status `PLANEJADA`.
- **Lógica de Importação Existente:**
  - Existe importação de CSV (relatório de máquinas), que atualiza O.S. com base em cruzamentos ("Universal Import Engine"), e fica nas configurações da empresa.
- **Lógica de Registro de Aplicação Relacionada:**
  - O.S. (ordens_servico) têm um array de `operacoes_multiplas` com produtos e dosagens.
  - A rotina de reconciliação de relatórios (o Universal Import Engine) relaciona execução com O.S. aberta e pode fechar a O.S. como CONCLUIDA ou DIVERGENTE baseada numa tolerância de área e doses.
- **Armazenamento/Cache/Sincronização:**
  - Coleção: `ordens_servico` no Firestore.
  - Sincronização offline: Usa o padrão do app com IndexedDB (gerenciado por `OfflineDB` em `docs/app.js`) e sincronização (via `App.data.addDocument` / `App.data.updateDocument` e cache no `App.state.ordens_servico`).

## 2. Identificação
- **Arquivos Principais:**
  - `docs/index.html`: Define as seções `#ordemServicoManual` e `#ordemServicoEscritorio`.
  - `docs/app.js`: Contém toda a lógica principal, gerenciamento de estado (`App.state`), ações (`App.actions.osManual`), UI (`App.osEscritorio`), integração de dados (`App.data`), sync offline (`OfflineDB`).
- **Funções Principais:**
  - `App.actions.osManual.generateOS()`: Criação/Edição de O.S.
  - `App.osEscritorio.renderList()` e associadas: Renderização da lista.
  - Funções no "Universal Import Engine" que reconciliam ordens em aberto (`App.reports.importMachineReport` etc. - *assumido com base na memória*).
- **Status Atuais:**
  - `PLANEJADA`, `EM_EXECUCAO`, `CONCLUIDA`, `DIVERGENTE`.
- **Pontos de Encaixe:**
  - Novo módulo de planejamento (`planejamento_os`) no menu sob Ordens de Serviço.
  - Alterações no modelo de `ordens_servico` durante a criação e atualização (no `generateOS`).
  - Rotinas de reconciliação para match avançado e timer.

## 3. Riscos de Compatibilidade
- **Sync/Offline:** O projeto usa sincronização offline própria com IndexedDB. Mudanças de esquema nas collections precisam ser refletidas nas listas `App.state` e rotinas de sync. É essencial adicionar novas "collections" ao array no `OfflineDB.init` se criarmos coleções novas (como `planejamento_os`, `apontamentos_os_importados`, etc.).
- **Regras atuais:** O motor de importação existente (Universal Import Engine) já lida com conciliação parcial e fechamento. O novo sistema não deve conflitar com ou quebrar o antigo.
- **Tipagem solta:** Firestore legacy pode ter `os_numero` como Number em vez de String. Novos campos precisam ser tratados com cautela e valores nulos ou indefinidos bem gerenciados.
